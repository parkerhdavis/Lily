# Troubleshooting

Common issues and solutions.

---


## Installation Issues

### Windows: SmartScreen blocks installation

**Symptom:** Windows shows "Windows protected your PC" message.

**Solution:** Click "More info" then "Run anyway." This is normal for unsigned applications.

**Alternative:** Right-click the installer, select Properties, check "Unblock," and click OK.

---

### Linux: AppImage won't run

**Symptom:** Error about FUSE or permission denied.

**Solution:**

```bash
# Make executable
chmod +x Lily_*.AppImage

# Install FUSE if needed
# Ubuntu/Debian:
sudo apt install fuse libfuse2
# Fedora:
sudo dnf install fuse fuse-libs
```

---

### macOS: "App cannot be opened"

**Symptom:** macOS blocks Lily with a security warning.

**Solution:** Right-click the app, select "Open," and confirm in the dialog. This only needs to be done once.

## Template Issues

### A variable isn't being detected

**Symptom:** A `{Variable}` placeholder in the template doesn't appear in the Variable Editor.

**Possible causes:**
1. Word split the variable across multiple internal XML elements
2. The curly braces are "smart" (typographic) quotes instead of plain braces

**Solutions:**
- Delete the placeholder in Word and retype it in one continuous action — don't pause, click away, or let spellcheck intervene mid-variable
- Make sure you're using plain curly braces `{ }`, not typographic braces `{ }` (disable smart quotes in Word if needed)
- Check for invisible formatting changes by selecting the entire placeholder and applying a single font/style

---

### Conditional variable shows a warning badge

**Symptom:** A variable shows a yellow warning badge with `!` in the Variable Editor.

**Cause:** The variable contains `??` or `::` characters but wasn't parsed as a conditional variable. This usually means Word split the variable syntax across XML elements.

**Solution:** Open the template in Word, delete the conditional variable placeholder, and retype the entire thing — including the `??` and `::` operators — in one go.

---

### Template formatting looks different in the preview

**Symptom:** The preview doesn't match what you see in Word.

**Explanation:** The preview is rendered as HTML, not in Word's layout engine. Minor differences in line wrapping, exact page breaks, and some advanced formatting are expected. The saved `.docx` file is always the authoritative version.

See [The Document Preview](wiki/03_Documents/02.02_The%20Document%20Preview.md) for details on what the preview supports.

## Document Issues

### Changes aren't saving

**Symptom:** Press `Ctrl+S` but changes don't persist.

**Check:**
- Is the client folder writable? Check file permissions.
- Is the `.docx` file open in another program (Word, LibreOffice)? Close it and try again.
- Is the disk full?

---

### "Unsaved" badge won't go away

**Symptom:** The "Unsaved" indicator stays visible after saving.

**Solution:** Try saving again with `Ctrl+S`. If the issue persists, check the application logs (see [App Settings](wiki/06_Settings/05.00_App%20Settings.md) for the log file location) for error messages.

---

### Multiple .lily files warning

**Symptom:** A warning appears when opening a client folder about multiple `.lily` files.

**Cause:** The client folder contains more than one `.lily` file, which can happen if files were manually copied or renamed.

**Solution:** Open the client folder in your file manager and remove the extra `.lily` file(s), keeping only the one you want.

---


## Performance Issues

### App is slow or unresponsive

**Possible causes:**
1. Very large document with many variables
2. Many documents in the client folder

**Solutions:**
- Close and reopen the client folder
- If a specific document is slow, check its size — very large `.docx` files (many pages, embedded images) take longer to process

## Getting More Help

### Application Logs

Lily writes daily log files that can help diagnose issues:

- **Windows:** `%APPDATA%\lily\logs\`
- **Linux:** `~/.config/lily/logs/`
- **macOS:** `~/Library/Application Support/lily/logs/`

Include relevant log entries when reporting a bug.

### Reporting a Bug

Email dev@parkerhdavis.com with:
- What you were trying to do
- What happened instead
- Your operating system and Lily version
- Relevant log entries (if available)
